The idea

A workstream is one feature split into many small units of work, each living in its own throwaway git worktree.

When a feature is too big for a single branch, you break it into units — a slice, a follow-up, a stacked layer. Each unit gets an isolated git worktree and branch of its own, so units never step on each other and can be built, reviewed, and merged independently. The worktree is a pure code checkout: a scratch space you can create, abandon, and recreate at will.

The catch is that a checkout can vanish — dropped, garbage-collected, or replaced — so nothing you care about keeping can live inside it. Every durable datum — a unit's progress and follow-ups, its history of decisions and restacks, and its identity map from unit to repo and branch — lives in a single global store under ~/.claude/workstreams/, one directory per workstream, shared across repos. Volatile facts that git and GitHub already own (current branch, base, PR number, status) are never copied in; they are derived live.

~/.claude/workstreams/<ws-id>/
  workstream.md          # metadata only
  units.md               # append-only ledger (unit ↔ repo/branch identity map)
  units/<unit-id>/
    progress.md          # MUTABLE current state: Tasks + Follow-ups checklists
    log.md               # APPEND-ONLY: created, dropped, restack, decision, note
Golden rule. The worktree holds only code. Drop and recreate it freely — all progress, history, and identity survive in the store.

Quickstart

Five minutes from nothing to a fresh agent building your first unit in its own worktree — then a board to watch it and a one-command way back in.

The ws-* suite tracks multi-unit work in a durable, cross-repo store at ~/.claude/workstreams/. A workstream is the container; each unit is one shippable slice of work with its own worktree and branch. Worktrees are disposable — all durable state lives in the store, so you can drop and recreate a checkout without losing progress.

Golden rule. Worktrees hold code only. Never write store files (progress.md, log.md) into a worktree — the store, not the checkout, is the source of truth.
  1. Create a workstream with ws-init. Give it a name; it computes a dated ws-id, scaffolds the store, and reports the id you'll pass to every later command. It creates no units or worktrees — just the container.
  2. Spin the first unit with ws-start <ws-id> "<what>". It mints a globally-unique unit-id (<ws-id>:<slug>), cuts a <slug> branch off the base, and creates a worktree via the active worktree-management flavor (a per-unit window is optional if the flavor defines one).
  3. Plan and build the unit with ws-resume — from the current session or the unit's worktree. It reads the charter + design, plans tasks as T1.. into progress.md, executes, and appends decisions to log.md.
  4. Check status anytime with ws-board. It's read-only: it reads the ledger, derives each unit's status live from git and GitHub, and renders a copy-paste-ready board grouped by state.
  5. Pick work back up with ws-resume <unit-id> — it finds the unit's worktree, reconciles any base drift, and drops you back into context.
ws-init"<name>"

Use when you need a new workstream and none exists yet.

Computes ws-id = <YYYY-MM-DD>-<slug(name)>, creates workstream.md plus an empty ledger, and reports the id. Give it more than one word so it can set the goal without asking.
$ /ws-init task templates
→ created workstream 2026-07-03-task-templates
  next: ws-start 2026-07-03-task-templates "<what the first unit does>"
ws-start<ws-id> "<what>" [--base <unit-id|branch>]

Use when starting a new unit of work in an existing workstream.

Resolves the workstream, mints unit-id = <ws-id>:<slug>, and creates the <slug> branch and worktree off the base via the active worktree-management flavor's create. It appends the ledger line and seeds progress.md and log.md; run ws-resume next to plan and build. Pass --base <unit-id> to stack on another unit's branch (its repo is inherited); --repo <org/repo> overrides the cwd repo. Omit <ws-id> when exactly one workstream exists.
$ /ws-start 2026-07-03-task-templates "template list page"

ws-resume takes over from here — no bootstrap prompt is printed. It reconstructs the unit's intent from its charter.md and the workstream's design: spec, plans with writing-plans (tasks as T1.. in progress.md), then executes with subagent-driven-development, appending decisions to log.md. Store files never live in the worktree.

Optional window. Run ws-resume in the current session, or — if your worktree-management flavor defines a window op — open a per-unit window for parallel work; if that window opens empty, launch claude there yourself.
ws-board[ws-id] [unit-id]

Use when you want to see or share where a workstream stands.

Reads the ledger and derives every unit's status live — no stored status, ever. Status is first-match-wins: dropped, then merged, then in-review (PR ready), then building (draft or no PR). Pass a unit-id to print that unit's full checklist plus recent log notes.
$ /ws-board 2026-07-03-task-templates
*Task templates* — 0/1 units done

🔄 In progress   • template list page — #4160 (draft) · 2/5 tasks
ws-resume<unit-id>

Use when resuming an existing unit, or picking work back up from inside its worktree.

Locates the unit's worktree via its ledger branch, reconciles any base drift (GitHub auto-retargets when a base PR merges), and restores you into the unit's live task state.
$ /ws-resume template-list-page
Tip. Everything after step 1 keys off the ws-id and unit-id. If a single workstream exists, most commands let you drop the ws-id and infer it.

The eight commands

Eight skills cover the full life of a unit of work — from spinning up a container to tearing a branch down, plus deciding what's next and configuring flavors — while every durable fact lives in the global store, never in a worktree.

Golden rule. Worktrees are disposable code checkouts; all durable state lives in the store. Every command reads the shared contract at ~/.claude/skills/ws-shared/SPEC.md before it acts, and never stores what git or GitHub already owns (branch, base, PR number, status — all derived live).
ws-init<workstream name>

Use when you need a new workstream and none exists yet — before starting any unit with ws-start.

Computes the ws-id (<YYYY-MM-DD>-<slug>) and creates the store container: workstream.md metadata, an empty append-only units.md ledger, and the units/ directory. Sets up the container only — it creates no unit and no worktree. 
$ /ws-init task templates
ws-start<ws-id> <what this unit does> [--base <unit-id|branch>]

Use when starting a new unit of work in an existing workstream — its own worktree plus ledger entry. Run ws-init first if no workstream exists.

Mints unit-id = <ws-id>:<slug>, resolves the repo (--repo › a --base unit's repo › cwd) and the base, then creates the <slug> branch and worktree off the base via the active worktree-management flavor's create. Appends the ledger line and seeds progress.md and log.md; ws-resume plans and builds. 
$ /ws-start 2026-07-03-task-templates "add is_template flag" --base schema-migration
ws-resume[unit-id]

Use when resuming or continuing an existing unit, or picking work back up from inside its worktree.

With no argument, infers the unit from the current worktree's branch. Ensures the worktree exists (reopening or recreating it if gone), reconciles the base against the recorded one and realigns on drift, then loads progress.md and log.md and continues at the first unchecked task. 
$ /ws-resume add-is-template-flag
ws-board[ws-id] [unit-id]

Use when you want to see or share where a workstream stands — "show the board", "what's done", "workstream status".

Reads the ledger, derives each unit's status and task counts, and renders copy-paste-ready markdown grouped by state (Done, In progress, Not started, Dropped) with open follow-ups listed across units. Given a unit-id, prints that unit's full checklist and recent notes. 
$ /ws-board 2026-07-03-task-templates
Tip. ws-board is strictly read-only — it derives everything and reconciles nothing. Reach for it freely to snapshot progress without touching git or GitHub.
ws-restack<unit-id> <new-base>

Use when a unit's base PR merged and its branch must move onto a new base, or GitHub auto-retargeted a dependent PR and the local branch needs realigning.

Resolves <new-base> (a branch, or another unit-id → that unit's branch) and rebases per the SPEC reconciliation. When we initiate — the PR's baseRefName is unchanged remotely — it runs gh pr edit --base first; in the auto case it skips that. Stops and reports on conflicts. 
$ /ws-restack add-is-template-flag master
ws-drop<unit-id>

Use when you want to drop, tear down, abandon, or clean up a unit's worktree and branch.

Shows exactly what will be removed and requires explicit confirmation, then removes the worktree and window and deletes the local branch. Appends a dropped log line but keeps progress.md and the ledger line as the reconciliation record. 
$ /ws-drop add-is-template-flag
Gotcha. ws-drop never deletes the remote branch or closes the PR unless you ask, and it preserves the store. To restart, run ws-start with the same intent — it versions the id with a -N suffix and records restart-of automatically.
ws-next[ws-id]

Use when you're unsure which ws-* command or which unit to act on next — after finishing a unit, when a PR merges, or any "what now?" moment.

Reads the ledger and each unit's live status, then names the single next action and the command to run it — it decides, it doesn't do the work (ws-resume does). Read-only, like ws-board. 
$ /ws-next 2026-07-03-task-templates
ws-config[show | set <group> <flavor> | set-overrides <path> | list [group] | add <group> <flavor>]

Use when viewing or changing workstream flavors — which external tool backs each behavior group.

Edits only ~/.claude/workstreams/flavors.ini — never a worktree. show prints the active flavor per group and which layers are in play; set switches one group's flavor; add scaffolds a custom flavor stub; set-overrides points at an external overrides file. See the Flavors section. 
$ /ws-config set worktree-management wmx

The durable model

Worktrees are disposable code checkouts; every durable fact lives in the store, and everything volatile is derived live from git and GitHub.

The suite splits every fact about a unit into two camps: what git and GitHub already own, and what only the workstream tracks. It never stores the former — branch, base, PR number, and status are read on demand, so the board can never drift out of sync with reality.

DatumWhere it livesNote
Current branchgit (the worktree)git rev-parse --abbrev-ref HEAD — never stored
Base / did GitHub retargetGitHubgh pr view --json baseRefName — never stored
PR number + draft/ready/mergedGitHubgh pr view --json number,isDraft,state — never stored
StatusderivedComputed live from the log + PR state (see precedence below)
Unit ↔ repo/branch identityunits.md ledgerAppend-only; set once at ws-start, prior lines never edited
Tasks + completion, follow-upsprogress.mdThe only home; mutable work-state SoT
Decisions, notes, drop, restack historylog.mdAppend-only history

Status is never written down — it is resolved on every read by walking four rules and taking the first match:

  1. A dropped line in log.md → dropped
  2. PR merged → merged
  3. PR ready → in-review
  4. PR draft, or no PR → building

Mutable state vs append-only history

The two per-unit files answer different questions and never overlap. progress.md is the mutable work-state source of truth — its Tasks and Follow-ups checklists always reflect the present, and task/follow-up ids (T1, F1) are monotonic per unit and never reused, even after check-off or removal. log.md is append-only history — created, dropped, restack, decision, note — a permanent record you add to but never rewrite.

~/.claude/workstreams/<ws-id>/
  workstream.md          # metadata only
  units.md               # append-only ledger (unit ↔ repo/branch)
  units/<unit-id>/
    progress.md          # MUTABLE work-state SoT — Tasks + Follow-ups
    log.md               # APPEND-ONLY history
The invariant. The log never stores current state; progress never stores history; nothing volatile — branch, base, PR, status — is stored at all. Derive it live.

IDs & base conventions

Identity is deterministic, so the same intent always resolves to the same place:

  1. ws-id = <YYYY-MM-DD>-<slug(name)> (from date -u +%Y-%m-%d) — this is the store directory name.
  2. unit-id = <ws-id>:<slug(what)>, globally unique by construction. A bare <slug> works as shorthand — resolved across all workstreams (one match wins; several prompt for the <ws-id>: prefix). Re-running the same intent takes a -N suffix and records restart-of (restart only).
  3. branch = <slug> unless the caller supplies one; disambiguated with -N on a git-branch collision in the target repo — including a slug that matches the base branch (: is illegal in refnames, so the branch is not the canonical id).
  4. base = the repo's default branch, resolved via gh repo view --json defaultBranchRef -q .defaultBranchRef.name — unless a base is supplied. A supplied base may name another unit-id, resolving to that unit's branch (stacking).
  5. repo (ws-start) resolves by precedence: explicit --repo › a --base unit's repo › the cwd repo. An explicit --repo that contradicts a --base unit's repo is an error.
Recorded base, not live base. A unit's recorded base is the base on its last created or restack line in log.md — never the live PR baseRefName, which GitHub may have moved when a base PR merged. Restack reconciles the two.

Flavors

External tools are pluggable. The skills never hardwire a worktree tool, a planning methodology, or a forge — each is a flavor you choose, so the same ws-* commands drive whatever you've configured.

A group is a fixed behavior category the skills define; a flavor is one implementation of it; an operation is a named slot a flavor fills with a one-line instruction (a shell command, or a skill to invoke). Exactly one flavor per group is active, globally. Swapping a flavor changes only mechanism — the ws bookkeeping (progress, log, ledger, PR-ready) is intrinsic and never moves.

GroupOperationsBuilt-in flavors (default first)
worktree-managementcreate · remove · locate · open-window (optional)git-worktree, wmx
spec-driven-developmentplan · execute · shipnone, superpowers
forgedefault-branch · pr-status · pr-create · pr-ready · pr-retargetgh

Your selection and any custom flavors live in ~/.claude/workstreams/flavors.ini, merged over the plugin's built-in defs; an optional overrides file layers on top. View or change all of it with ws-config.

Configure with /ws-config. ws-config show prints the active flavor per group; ws-config set <group> <flavor> switches one; ws-config add scaffolds a custom flavor to fill in. gh is the assumed forge baseline — a non-GitHub user adds a custom forge flavor via the overrides file.

Runbooks

Five recipes for the everyday shape of workstream work — running units side by side, picking one up cold, restarting a dead-end, restacking after a merge, and sharing status.

Golden rule. Worktrees are disposable code checkouts; all durable state lives in the store at ~/.claude/workstreams/<ws-id>/. Drop and recreate worktrees freely — progress survives because it was never stored in them.

1 · Run two units in parallel

Each unit gets its own worktree off a shared store, so two units run at once without stepping on each other — open a per-unit window if you want them side by side. Start them from the same ws-id; the ledger and per-unit progress.md keep them separate.

ws-start<ws-id> "<what>" [--base <unit-id|branch>] [--repo <org/repo>]

Use when starting a new unit in an existing workstream — its own worktree and ledger entry.

Run it once per unit. Each call creates the <slug> branch and worktree via the active worktree-management flavor, appends a ledger line, and seeds that unit's progress.md + log.md. Units share only the store dir — code stays isolated per worktree.
$ /ws-start 2026-07-03-task-templates "template CRUD api"
$ /ws-start 2026-07-03-task-templates "template picker ui"
Tip. To stack the second unit on the first instead of running truly parallel, pass --base <unit-id>; it resolves to that unit's branch and records stacked-on on the ledger line.

2 · Resume a unit cold — new session or next day

Nothing is carried in your head or the shell: ws-resume reads the store and reconstructs the unit. Run it from inside the worktree (id inferred from the branch) or name the unit explicitly.

ws-resume[unit-id]

Use when picking work back up — a fresh session, the next day, or after the worktree was dropped.

  1. Resolves the unit via the bare-slug resolver → ws-id, repo, branch (or infers it from the current branch).
  2. Self-locates the worktree: reuses it in place, cds in if you're elsewhere, reopens via the active worktree-management flavor if the branch survives, or fresh-starts off the default branch using stored progress as the baseline if the branch is gone.
  3. Reconciles the base if the live PR base drifted from the recorded base (see recipe 4).
  4. Loads progress.md + recent log.md, runs git log -5 and verification, then continues at the first unchecked task.
$ /ws-resume template-crud-api

3 · Drop and restart from scratch

Tear the worktree down, then re-run ws-start with the same intent — the id versions to -2 and the old unit is kept as history.

ws-drop<unit-id>

Use when abandoning a unit's worktree and local branch — a dead-end approach you want to redo cleanly.

Shows exactly what it will remove and requires confirmation, then removes the worktree/window and the local branch. It appends a dropped log line but keeps progress.md and the ledger line as the reconciliation record. The remote branch and PR stay unless you ask.
$ /ws-drop template-crud-api
ws-start<ws-id> "<same what>"

Use when restarting the dropped unit fresh.

Re-running with an intent whose slug already exists takes the next -N suffix and records restart-of=<original-unit-id> on the new ledger line — history stays intact.
$ /ws-start 2026-07-03-task-templates "template CRUD api"
  # → unit template-crud-api-2  (restart-of=template-crud-api)

4 · Restack after a base PR merges

When a unit is stacked on another and that base merges, the dependent branch must move onto its new base. A unit's recorded base is the base on its last created/restack log line — never the live PR base, which GitHub may have moved.

ws-restack<unit-id> <new-base>

Use when a unit's base PR merged, or GitHub auto-retargeted a dependent PR and the local branch needs realigning.

Rebases onto the recorded base via the pinned command, then appends a restack line. <new-base> may be a branch or another unit-id (resolved to its branch).
OLD=$(git merge-base HEAD origin/<recorded-base>)
git fetch
git rebase --onto origin/<new-base> $OLD
$ /ws-restack template-picker-ui template-crud-api
Auto vs we-initiate gate. Compare gh pr view --json baseRefName to the recorded base. If it is unchanged remotely, we initiate — run gh pr edit <pr> --base <new-base> first, then rebase. If it has already changed (GitHub auto-retargeted when the base PR merged), skip the gh pr edit and just rebase. On conflicts, stop and report.
Tip. Only ws-restack (explicit) and ws-resume (on detecting drift) reconcile. ws-board is read-only and never rebases.

5 · Share status

Print a copy-paste-ready markdown board grouped by state — done, in progress, not started, dropped — with per-unit task counts and open follow-ups.

ws-board[ws-id] [unit-id]

Use when you want to see or share where a workstream stands — "show the board", "what's done".

Derives every status live (nothing stored) and renders markdown you can paste into Slack. With a unit-id, it prints that unit's full progress.md checklist plus recent log.md notes.
*Task templates* — 1/3 units done

✅ Done          • template CRUD api — #4160
🔄 In progress   • template picker ui — #4162 (draft) · 2/5 tasks
⏳ Not started   • bulk import   (deps: template-crud-api)
$ /ws-board 2026-07-03-task-templates

Cheat sheet

Eight skills, one durable store: worktrees hold code, the store holds truth, and status is always derived live from git and GitHub.

ws-startStart a unit: its own worktree, branch, ledger line, and progress.md/log.md.

├── workstream.md          # metadata only (id, name, goal, design)
├── units.md               # append-only ledger: unit ↔ repo/branch identity
└── units/
    └── <unit-id>/
        ├── progress.md     # MUTABLE work-state: Tasks + Follow-ups checklists
        └── log.md          # APPEND-ONLY: created · dropped · restack · decision · note

ws-id = <YYYY-MM-DD>-<slug(name)> · unit-id = <ws-id>:<slug(what)> (bare <slug> shorthand; -N = restart) · branch = <slug>. The store is global, not per repo.

progress.md = work-state source of truth (mutable). Tasks and follow-ups live here and nowhere else. Ids (T1, F1) are monotonic per unit and never reused, even after check-off or removal. No history is kept here.
log.md = append-only. One line per event — created, dropped, restack, decision, note. Never edit or delete prior lines; the log never stores current state.
Nothing volatile is stored — derive it live. Current branch comes from git, base and retargeting from gh pr view --json baseRefName, PR number and draft/ready/merged from GitHub, and status from a first-match cascade: dropped → merged → in-review → building.

Gotchas & FAQ

Worktrees are disposable code checkouts; every durable fact lives in the global store or is derived live from git and GitHub — so most "I lost it" moments are recoverable.

Golden rule. Never store what git or GitHub owns. Branch, base, PR number, and status are always derived live; only tasks, follow-ups, and history are written to the store. If a worktree feels lost, the store still knows the unit.
QuestionAnswer
Q: I deleted the worktree — is my work lost? A: No. Worktrees are disposable; all durable state lives in ~/.claude/workstreams/<ws-id>/. Run /ws-resume <unit-id>: if the branch still exists it reopens the worktree; if the branch is also gone it starts fresh off the repo default, using the store's progress.md as the restart baseline.
Q: Which unit am I in? A: Run /ws-resume with no argument. It infers the unit from the current worktree's branch by scanning every units.md ledger in the store, then resolves the ws-id, repo, and branch for you.
Q: My base PR merged — how do I move onto the new base? A: Run /ws-restack <unit-id> <new-base>. It rebases --onto the new base from the recorded merge-base and appends a restack line to log.md. When you initiate and the PR base is unchanged remotely, it also runs gh pr edit --base first.
Q: GitHub auto-retargeted my dependent PR — do I still restack? A: The realign still happens, but the gh pr edit step is skipped since GitHub already moved the base. /ws-resume also auto-reconciles: if the live baseRefName differs from the recorded base, it realigns and records a restack line before continuing.
Q: I opened a per-unit window and it didn't auto-start the agent. A: Windows are optional — run ws-resume in your current session, or launch claude in the window yourself. A window only autostarts the agent when your worktree-management flavor is configured to.
Q: Can a workstream span multiple repos? A: Yes. The store is global, not per-repo. Each unit records its own repo=<org/repo> (and branch) on its ledger line in units.md, so units of one workstream can live in different repositories.
Optional window. A per-unit window is optional — run ws-resume in your current session instead. If you do open one and it doesn't start the agent, run claude manually (a window only autostarts when the worktree-management flavor is configured to).
Tip. Read-only by design: ws-board never reconciles a base. Only ws-restack (explicit) and ws-resume (on detecting drift) move a branch onto a new base.
